在不改变任何对外行为的前提下，对项目中的 **Project 管理功能模块（project）** 做重构优化，目标是提升可维护性与一致性。对外行为指：接口路径/入参出参结构/状态码与错误码语义/业务规则/数据库含义均保持不变。最终必须产出一份文档《project优化清单》（见输出要求）。

------

### 必做优化项（固定 4 项，全部完成）

1. **统一 DTO/VO 边界**
   - Controller 只接收 Request DTO、只返回 Response VO
   - Controller 禁止直接暴露/返回 Entity
   - Entity 仅用于持久化与内部领域模型
2. **抽取重复逻辑**
   - 将 Project 模块中重复的：参数校验触发点、Entity↔DTO/VO 映射、异常转换、日志打印 统一抽取到固定位置
   - 目标：Controller/Service 不再出现重复的手写映射与重复异常处理代码
3. **改善分层**
   - Controller 变薄：仅做参数接收/校验触发/调用 service/返回
   - Service 更清晰：表达业务流程，避免把业务判断散落在 Controller
   - Mapper/Repository 仅负责数据访问
4. **统一错误码与返回体构造**
   - Project 模块所有成功/失败响应的构造方式统一（集中封装构造器/工厂/工具类）
   - 错误码集中定义并复用（不允许散落 magic number）
   - 保持对外返回结构与语义不变

------

### 验收（必须全部满足）

- 功能无回退：Project 模块所有接口行为保持一致（包含：CRUD、分页、keyword 查询、逻辑删除、参数校验、统一返回体、统一异常）
- `mvn test` 必须通过
- 必须新增或更新测试 **≥3 个**，覆盖关键路径（建议覆盖：创建成功、参数校验失败、删除后查询不存在、分页 keyword 生效中的至少 3 条）
- 输出 3-5 条“重构收益”，必须具体可核验（例如：删除了哪些重复、边界更清晰、测试更容易写等）

------

### 约束（硬性）

- 不允许通过吞异常/绕过逻辑/临时 return 来“假重构”
- 不允许修改对外接口契约（路径、字段、返回结构、错误码语义）
- 重构范围聚焦 Project 模块及其直接相关代码，避免横向大重构

------

### 输出（按顺序，不能缺）

1. **关键变更文件清单**：新增/修改文件路径列表
2. **重构说明**：本次做了哪些结构调整（按上述 4 项分别写清）
3. **验证步骤**：
   - `mvn test` 命令 + 期望结果
   - 如有接口验证，给出 curl 示例 + 期望结果
4. **文档文件内容**：新增 `project优化清单.md`，必须包含：
   - 背景与目标（不改外部行为）
   - 4 项必做优化的落地结果（每项：改动点/涉及文件/前后对比要点）
   - 回归与测试覆盖说明（列出新增/更新测试）
   - 重构收益 3-5 条（可核验、具体）
